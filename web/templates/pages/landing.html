The blind spot in your monitoring

The problem

Most monitoring tools run checks from a single external server. If your service is behind a firewall, NAT, or VPC — they can't see it. You're flying blind on the infrastructure that matters most.

The WatchDog approach

WatchDog flips the model. A lightweight agent runs inside your network, checking services locally and reporting back to a central hub. No inbound ports. No VPN tunnels. No blind spots.

How it works

1. Deploy an agent

Single Go binary. Run it inside your network — a VPC, a homelab, behind a NAT. No config file needed.

2. Agent runs checks

HTTP, TCP, DNS, TLS, Docker, database, and system metrics. The hub pushes monitor configs to the agent over WebSocket. The agent reports back.

3. Get alerted

Three consecutive failures trigger an incident. Get notified via Discord, Slack, Email, Telegram, PagerDuty, or webhooks.

Features

What you get out of the box.

Agent-based monitoring

The agent runs inside your network, so it can reach things that external monitoring tools can't — internal databases, Docker containers, private APIs, services behind a firewall or NAT.

Live dashboard

Real-time status via SSE. Latency sparklines, uptime bars, system metric charts, and incident timeline. No page refresh.

Smart alerts

3-strike rule before firing. Supports Discord, Slack, Email, Telegram, PagerDuty, and webhooks.

Eight check types

HTTP with content validation, TCP, Ping, DNS, TLS certificate expiry tracking, Docker containers, database connectivity (PostgreSQL, MySQL, Redis), and system metrics with threshold alerts.

Zero config agent

The agent takes a hub URL and an API key. That's it. Monitor configs are pushed from the hub over WebSocket — no config files, no restarts.

$ ./watchdog-agent \
--hub https://hub.example.com \
--key wdg_a8f29c...
[connected] receiving 6 monitors...

Incident tracking

Incidents are created automatically on failure. Acknowledge, resolve, and track time-to-resolution. Full history per monitor.

Public status pages

Share real-time service health with your team or customers. 90-day uptime history, incident timeline, and custom URLs. No authentication required to view.

REST API

Token-authenticated API for programmatic access. Manage monitors, agents, and incidents from your own scripts or CI/CD pipelines.

Audit trail

Every login, monitor change, and incident action is logged with timestamps, user IDs, and IP addresses. Full event history for compliance and debugging.

Architecture

Three components. One binary each.

Hub

Central server. Stores configs, processes heartbeats, serves the dashboard, fires alerts. Go + PostgreSQL + TimescaleDB.

Agent

Lightweight Go binary that runs inside your network. Connects to the hub over WebSocket. Receives check configs, reports results.

Dashboard

Real-time web UI served by the hub. Live updates via SSE. Public status pages. HTMX + Alpine.js — no SPA framework.

Built with: Go PostgreSQL TimescaleDB WebSocket HTMX

Security by design

Your infrastructure credentials never leave your network.

Zero inbound ports

The agent connects outbound to the hub over WebSocket. No listening ports, no attack surface on your network.

Minimal attack surface

Single static Go binary. No runtime dependencies, no interpreters, no package managers. Agent fingerprinting detects impersonation.

Credentials stay local

Self-hosted hub means your database passwords, API keys, and service endpoints never leave your infrastructure.
